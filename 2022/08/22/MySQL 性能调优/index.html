MySQL 逻辑体系架构

在 Oracle 和 SQL Server 等数据库中，数据存储管理机制都是一样的。和其它数据库相比，MySQL 有点与众不同，它的架构可以在多种不同场景中应用，并发挥良好作用，用户可以根据自己的需求为数据表选择不同的存储引擎，也可以根据自己的需求编写自己的存储引擎，甚至一个库中不同的表使用不同的存储引擎 ( 存储引擎是针对表的 ) 。

连接层

  • 连接者：不同程序设计语言与 MySQL 的交互

服务层

  • 连接池： 创建并管理用户的连接
  • 管理服务和工具组件：MySQL 提供的系统管理和控制工具，例如备份恢复、MySQL 复制、集群等等
  • SQL 接口：接受用户的 SQL 命令，并返回用户需要的查询结果
  • 解析器：解析并验证 SQL 命令 （权限、语法结构）
  • 优化器：优化 SQL 查询命令
  • 缓存： 缓存查询结果，如果该查询有缓存结果，那么可以直接去缓存中取数据

引擎层

  • 可插拔式存储引擎：说白了就是如何管理操作数据（存储数据、如何更新、查询数据等）的一种方法

存储引擎

数据库存储引擎是数据库底层软件组织，数据库管理系统（DBMS）使用存储引擎进行创建、查询、更新和删除数据，不同的存储引擎提供不同的存储机制、索引、锁的粒度等功能，使用不同的存储引擎，还可以获得特定的功能。

MySQL 支持数 10 种存储引擎，但是在实际开发中，常常只用到 InnoDB 和 MyISAM。

InnoDB 和 MyISAM

对比项 MyISAM InnoDB

缓存 只缓存索引，不缓存真实数据 不仅缓存索引，还缓存真实数据，对内存要求较高，而且内存大小对性能有决定性的影响
表空间 小 大
关注点 性能 事务
默认安装 Y Y
  • 事务：InnoDB 是事务型的，可以使用 Commit 和 Rollback 语句，所以 InnoDB 可以通过事务的隔离级别保持数据的一致性
  • 并发：MyISAM 只支持表级锁，而 InnoDB 还支持行级锁，所以 InnoDB 的并发程度高，适合高并发的场景
  • 外键：InnoDB 支持外键
  • 缓存：MyISAM 只缓存索引，不缓存真实数据，它是非聚簇索引（.frm、.MYI、.MYD 索引和数据相分离），而 InnoDB 是聚簇索引（.frm、.idb 索引和数据不分离）
  • 备份：InnoDB 支持在线热备份
  • 崩溃恢复：MyISAM 崩溃后发生损坏的概率比 InnoDB 高很多，而且恢复的速度也更慢
  • 其它特性：MyISAM 支持压缩表和空间数据索引

存储层

  • 文件存储：主要是将数据存储在运行于裸设备的文件系统之上，并完成与存储引擎的交互

SQL 性能下降分析

原因分析

  • 查询语句写的烂
  • 未建立索引，或者建立了索引，但是索引没有生效
  • 关联查询太多 join on（老旧系统的设计缺陷或不得已的需求）
  • 服务器调优及各个参数设置（缓冲、线程数等）
  • 硬件问题

优化思路

  1. 服务器的状态监测：top、free、iostat 和 vmstat 来查看系统的性能状态，比如 CPU 和 磁盘 IO 的使用情况；
  2. 观察，至少跑 1 天，看看生产的慢 SQL 情况；
  3. 慢查询的开启并捕获，比如超过 3 秒钟的就是慢 SQL，并将它抓取出来；
  4. Explain + 慢 SQL 分析；
  5. show profile 查询 SQL 在 MySQL 服务器里面的执行细节和生命周期情况；
  6. 数据库服务器的参数调优。

索引简介

概念

索引（Index）是帮助 MySQL 高效获取数据的排好序的数据结构。

为什么索引查询数据快呢？因为数据是按某种数据结构进行存储的，比如 InnoDB 存储引擎默认的索引为 B+Tree 索引，B+Tree 为插入的数据进行排序，在 B+Tree 中查找一个数的平均时间复杂度为 O(log(m,N))，m 表示 B+Tree 的出度，N 表示 B+Tree 的节点数量。

一般来说索引本身也很大，不可能全部存储在内存中，因此索引往往以文件的形式存储在磁盘上。

索引的优缺点

优点：

  • 提高查询数据的效率，大大减少了服务器需要扫描的数据行数，降低了 IO 成本
  • 对数据进行排序 ，避免 filesort 和 创建临时表，降低数据排序的成本，减少了 CPU 的消耗

缺点：

  • 实际上索引也是一张表，该表保存了主键和索引字段，并指向实体表的记录，所以索引列也是要占用空间的
  • 虽然索引大大提高了查询效率，但是却会降低表的更新效率，因为在更新表的数据同时，还需要更新表的索引

索引分类

  • 单值索引：即一个索引只包含单个列，一个表可以有多个单列索引

  • 唯一索引：索引列的值必须唯一，但允许有空值

  • 复合索引：即一个索引包含多个列

  • 覆盖索引：要查询的列被所建的复合索引覆盖（索引包含所有需要查询的列），比如 select col1, col2，建立的索引是 idx_col1_col2_col3，那么就是覆盖索引。

    如果要使用覆盖索引，那么就要避免使用 select *，有人会说那我把所有的列都建立作为一个复合索引，那么我用 select * 也没事啊，但是你要知道，索引也是占用存储空间的，将所有的字段作为复合索引，会导致索引文件太大，查询性能下降。

    使用覆盖索引具有以下优点：

    • 索引通常远小于数据行的大小，只读取索引能大大减少数据访问量；
    • 一些存储引擎（例如 MyISAM）在内存中只缓存索引，而数据依赖于操作系统来缓存。因此，只访问索引可以不使用系统调用（通常比较费时）；
    • 对于 InnoDB 引擎，若辅助索引能够覆盖查询，则无需访问主索引，避免回表。

MySQL 索引的数据结构

B+Tree 索引

B+Tree 索引是 MySQL 的 InnoDB 存储引擎的默认索引类型。使用该索引检索数据不再需要对全表进行扫描，而是转换为对树的搜索。

B+Tree 是一棵多路平衡搜索树，它能够对新插入的数据进行排序，类似二叉树一样，小于根节点的数据在根的左边，大于根节点的数在根的右边，只不过 B+Tree 是多叉树；B+Tree 树所有的叶子节点位于同一层，叶子节点之间通过指针连接，形成双向链表。

B+Tree 索引适用于全键值、键值范围和键前缀查找，其中键前缀查找只适用于最左前缀查找。如果不是按照索引列的顺序进行查找，则无法使用索引。

InnoDB 的 B+Tree 索引分为主索引和辅助索引。主索引的叶子节点 data 域记录着完整的数据记录，这种索引方式被称为聚簇索引。因为无法把数据行存放在两个不同的地方，所以一个表只能有一个聚簇索引。

辅助索引的叶子节点的 data 域记录着主键值和索引列，因此在使用辅助索引进行查找时，需要先查找到主键值和索引列，然后再到主索引中进行查找，这个操作也叫做回表。

与红黑树的比较

红黑树等平衡树也可以用来实现索引，但是文件系统及数据库系统普遍采用 B+Tree 作为索引结构，这是因为使用 B+Tree 访问磁盘数据有更高的性能。

  1. B+Tree 有更低的树高

    平衡树的树高 O(h)=O(logdN)，其中 d 为每个节点的出度。红黑树的出度为 2，而 B+Tree 的出度一般都非常大，所以红黑树的树高 h 很明显比 B+Tree 大非常多。

  2. 磁盘访问原理

    操作系统一般将内存和磁盘分割成固定大小的块，每一块称为一页，内存与磁盘以页为单位交换数据。数据库系统将索引的一个节点的大小设置为页的大小，使得一次 I/O 就能完全载入一个节点。

    如果数据不在同一个磁盘块上，那么通常需要移动制动手臂进行寻道，而制动手臂因为其物理结构导致了移动效率低下，从而增加磁盘数据读取时间。 B+Tree 相对于红黑树有更低的树高，进行寻道的次数与树高成正比，在同一个磁盘块上进行访问只需要很短的磁盘旋转时间，所以 B+Tree 更适合磁盘数据的读取。

  3. 磁盘预读特性

    为了减少磁盘 I/O 操作，磁盘往往不是严格按需读取，而是每次都会预读。预读过程中，磁盘进行顺序读取，顺序读取不需要进行磁盘寻道，并且只需要很短的磁盘旋转时间，速度会非常快。并且可以利用预读特性，相邻的节点也能够被预先载入。

哈希索引

哈希索引能以 O(1) 时间进行查找，但是失去了有序性。

  • 无法用于排序与分组；
  • 只支持精确查找，无法用于部分查找和范围查找。

InnoDB 存储引擎有一个特殊的功能叫 “自适应哈希索引”，当某个索引值被使用的非常频繁时，InnoDB 会在 B+Tree 索引之上再创建一个哈希索引，这样就让 B+Tree 索引具有哈希索引的一些优点，比如快速的哈希查找。

全文索引

MyISAM 存储引擎支持全文索引，用于查找文本中的关键词，而不是直接比较是否相等。

查找条件使用 MATCH AGAINST，而不是普通的 WHERE。

全文索引使用倒排索引实现，它记录着关键词到其所在文档的映射。

InnoDB 存储引擎在 MySQL 5.6.4 版本中也开始支持全文索引。

空间数据索引

MyISAM 存储引擎支持空间数据索引（R-Tree），可以用于地理数据存储。空间数据索引会从所有维度来索引数据，可以有效地使用任意维度来进行组合查询。

必须使用 GIS 相关的函数来维护数据。

索引的使用场景

适用场景

  1. 主键自动建立唯一索引
  2. 频繁作为查询条件的字段应该创建索引
  3. 查询中与其他表关联的字段、外键关系建立索引
  4. 查询中排序的字段，排序字段若通过索引去访问将大大提高排序的速度
  5. 查询中统计或者分组的字段

不适用场景

  1. 频繁更新的字段不适合创建索引，因为每次更新不单单是更新记录，还会更新索引，加重 IO 负担

  2. where 条件里用不到的字段不创建索引

  3. 选择性弱的列不适合建立索引，如果某个数据列包含许多重复的内容（数据重复且平均分布），为它建立索引就没有太大的实际效果

    深刻理解选择性弱的列不适合建立索引，或者即使建立了，但是索引失效。like查询后置% 索引竟然也失效

  4. 对于非常小的表，大部分情况下简单的全表扫描比建立索引更高效

Explain 查看执行计划

概念

使用 Explain 关键字可以模拟优化器执行 SQL 查询语句，从而知道 MySQL 是如何处理你的 SQL 查询语句的。

作用

  • 表的读取顺序：id + table
  • 数据查询的类型：select_type
  • 查询的级别：type
  • 哪些索引可能被使用：possible_keys
  • 哪些索引实际被使用：key
  • 表之间的引用：id + table + select_type
  • 每张表有多少行被优化器查询：rows

字段解释

用法：Explain + SQL 查询语句，可以得到以下信息。

id

显示查询的序列号，包含一组数字，表示查询中执行 select 子句或操作表的顺序。

  1. id 相同，执行顺序由上往下
  1. id 不同，如果是子查询，id 的序号会递增，id 值越大优先级越高，越先被执行

  1. id 相同和不同，同时存在。id 如果相同，可以认为是一组，从上往下顺序执行；id 值越大优先级越高，越先被执行

    table：derived 表示它是一张衍生表，也称为虚表。

select_type

显示查询的类型，主要用于区别普通查询、联合查询、子查询等复杂查询。共有以下六种类型：

  • SIMPLE：简单的 select 查询，查询中不包含子查询或者 UNION
  • PRIMARY：联合查询，查询中若包含任何子查询，最外层查询则被标记为 PRIMARY
  • SUBQUERY：子查询，在 select 或者 where 列表中包含了子查询
  • DERIVED：在 from 列表中包含的子查询被标记为 DERIVED（衍生），MySQL 会递归执行这些子查询，把结果放在临时表里
  • UNION：若第二个 select 出现在 UNION 之后，则被标记为 UNION；若 UNION 包含在 from 子句的子查询中，外层 select 将被标记为 DERIVED
  • UNION RESULT：从 UNION 表获取结果的 select，它的 id 为 NULL

table

显示查询与哪张表的有关。

type

显示查询的级别，从最好到最差依次是：system > const > eq_ref > ref > range > index > ALL，一般来说，至少得保证查询达到 range 级别，最好达到 ref。

  • system：表中只有一行记录（等于系统表），这是 const 类型的特例，一般不会出现。

  • const：常量索引扫描，表示通过索引查找一次就找到了对应的值。常见于主键或唯一索引，因为只匹配一行数据，所以很快。如将主键索引置于 where 条件中，MySQL 会将该 where 条件转换为一个常量。

  • eq_ref：唯一性索引扫描，对于每个索引键，表中只有一条记录与之匹配，常见于主键或唯一索引扫描。
  • ref：非唯一性索引扫描，和 eq_ref 不同，ref 会返回符合条件的所有行，也可能只有一行，条件列非主键或唯一索引。
  • range：使用索引检索给定范围的行。一般就是在你的 where 语句中出现了 between、<、>、in 等关键字的查询。
  • index：全索引扫描，使用了索引但是没用通过索引进行过滤，一般是使用了覆盖索引或者是利用索引进行了排序分组。index 与 ALL 区别为 index 类型只遍历索引树，这通常比 ALL 快，因为索引文件通常比数据文件小。也就是说虽然 ALL 和 index 都是读全表，但 index 是从索引文件中读取的，而 ALL 是从数据文件中读取。
  • ALL：全表扫描

possible_keys

  1. 显示查询可能使用的索引，一个或多个；
  2. 若查询涉及的字段上存在索引，则该索引将被列出，但不一定被查询实际使用。

key

  1. 显示查询实际使用的索引，如果为 NULL 则没有使用索引；

  2. 若查询中使用了覆盖索引，则该索引只出现在 key 列表中，如下图所示。

key_len

显示索引中使用的字节数，可通过 key_len 计算查询中使用的索引的长度。key_len 字段能够帮你检查是否充分的利用上了索引，ken_len 越长，说明索引使用的越充分，在不损失精度的情况下，长度越短越好。

ref

显示索引匹配了哪个库的哪张表的哪一列或者常量，那些列或常量被用于查找索引列上的值。

rows

显示查询扫描的数据行数，越少越好。

Extra

包含不适合在其他列显示，但十分重要的额外信息。

  • Using filesort：说明 MySQL 会对数据使用一个外部的索引排序，而不是按照表内的索引顺序进行排序，这种排序称为 “文件内部排序” ，需要优化。
  • Using temporary：说明 MySQL 在对查询结果排序时使用了临时表，常见于排序 order by 和分组查询 group by，必须优化，很有可能就是它拖慢了你的 SQL。

  • Using index：表示相应的 select 操作中使用了覆盖索引（Coveing Index），避免全表扫描，效率不错！
    如果同时出现 using where，说明索引被用来执行索引键值的查找；
    如果没有出现 using where，说明索引用来读取数据而非执行查找动作。

  • Using where：此查询使用了 where 过滤

  • Using join buffer：此查询使用了连接缓存

  • impossible where：where 子句的值总是 false，不能用来获取任何元组

  • select tables optimized away：在没有 group by 子句的情况下，基于索引优化 MIN/MAX 操作或者对于 MyISAM 存储引擎优化 COUNT(*) 操作，不必等到执行阶段再进行计算，查询执行计划生成的阶段即完成优化。

  • distinct：优化 distinct，在找到第一匹配的元组后即停止找同样值的工作

索引优化策略

索引单表、双表、三表优化

  1. 尽可能减少 Join 语句中的 NestedLoop 的循环总次数，永远用小结果集驱动大的结果集；
  2. 优先优化 NestedLoop 的内层循环；
  3. 保证 Join 语句中被驱动表上 Join 条件字段已经被索引，left join 索引加右表，right join 索引加左表；
  4. 当无法保证被驱动表的 Join 条件字段被索引，并且内存资源充足的前提下，不要太吝惜 JoinBuffer 的设置。

索引使用口诀

全值匹配我最爱，最左前缀要遵守；
带头大哥不能死，中间兄弟不能断；
索引列上少计算，范围之后全失效；
LIKE 百分写最右，覆盖索引不写星；
不等空值还有 or，索引失效要少用；
VARCHAR 引号不可丢，SQL 高级也不难！

索引使用策略

  1. 最佳左前缀法则，如果索引了多列，要遵守最左前缀法则，即查询从索引的最左列开始，并且不要跳过索引中间的列
  2. 在索引列上做任何操作（计算、函数、自动或手动类型转换）会导致索引失效而转向全表扫描
  3. 存储引擎不能使用索引中范围条件右边的列
  4. like 以通配符开头（‘$abc…’ 或者 ‘%abc…’）会导致索引失效变成全表扫描操作，可以使用覆盖索引解决 ‘%abc…’ 索引失效的问题
  5. 尽量使用覆盖索引，因为覆盖索引只访问索引文件就得到了查询结果，不需要回表
  6. 在使用不等于（!= 或者 <>）的时候无法使用索引，会导致全表扫描
  7. is null、is not null 也无法使用索引
  8. 少用 or，用它连接时会索引失效
  9. 字符串不加单引号索引失效，因为 MySQL 会自动进行类型转换
  10. order by 的情况很特殊

索引使用示例

假设 index(a, b, c)：

where 语句 索引是否被使用
where a =3 Y，使用到 a
where a =3 and b =5 Y，使用到 a，b
where a =3 and b =5 and c =4 Y，使用到 a，b，c
where b =3 或者 where b=3 and c=4 或者 where c=4 N
where a =3 and c=5 使用到 a ，但是 c 不可以，因为 b 中间断了
where a =3 and b>4 and c=5 使用到 a 和 b，c 不能用在范围之后，因为 b 断了
where a =3 and b like ‘kk%’ and c =4 Y，使用到 a，b，c
where a =3 and b like ‘%kk’ and c =4 Y，只用到 a
where a =3 and b like ‘%kk%’ and c =4 Y，只用到 a
where a =3 and b like ‘k%kk%’ and c =4 Y，使用到 a，b，c

其他优化策略

小表驱动大表

因为小表需要全表扫描，大表可以使用索引进行查询或者排序，所以不要将数据多的表作为小表。

SELECT……FROM table WHERE EXISTS(subquery)
该语法可以理解为：将主查询的数据，放到子查询中做条件验证，根据验证结果（True 或 False）来决定主查询的数据结果是否可以保留。

order by 关键字排序优化

  1. ORDER BY 满足以下任意条件时，会使用 Index 方式排序：
    • ORDER BY 语句使用索引最左前列；
    • 使用 where 子句与 Order By 子句条件列的组合满足索引最左前列，需要 where 索引列条件为常量。
  2. 如果不在索引列上，filesort 有两种算法：双路排序和单路排序
    • 双路排序
      • MySQL4.1 之前使用双路排序，需要两次磁盘 IO 才能得到结果；
      • 第一次磁盘 IO 取出要排序的字段和行号，在 sort_buffer 中进行排序；第二次磁盘 IO 取出全部字段，这时候是已经排好序的。
    • 单路排序
      • 对双路排序的改进，只需要一次磁盘 IO 就可以得到结果；
      • 第一次磁盘 IO 取出要排序的字段和其他要查询的所有字段，在 sort_buffer 中进行排序；
      • 它的效率更快一些，避免了第二次读取数据，并且把随机 IO 变成顺序 IO，但是它会使用更多的空间，因为它把每一行都保存在内存中了，所以在必要的时候，需要调整 sort_buffer 的大小。
  3. 使用 order by 时 select *是一个大忌，只查询需要的字段，这点非常重要。在这里的影响是:
    • 当查询的字段大小总和小于 max_length_for_sort_data，而且排序字段不是 TEXT/BLOB 类型时，会用改进后的算法——单路排序，否则用老算法——多路排序
    • 两种算法的数据都有可能超出 sort_buffer 的容量，超出之后，会创建 tmp 文件进行合并排序，导致多次 IO，但是用单路排序算法的风险会更大一些，所以要提高 sort_buffer_size
  4. 优化策略
    • 增大 sort_buffer_size 参数的设置
      不管用哪种算法，提高这个参数都会提高效率，但是要根据系统的能力去提高，因为这个参数是针对每个进程的。
    • 增大 max_length_for_sort_data 参数的设置
      提高这个参数，会增加用单路排序算法的概率。但是如果设的太高，数据总容量超出 sort_buffer_size 的概率就增大，明显症状是高的磁盘 I/O 活动和低的处理器使用率。

使用示例

假设 index(a, b, c)：

order by 语句 是否使用索引排序
order by a Y，order by 能使用索引最左前缀
order by a, b 同上
order by a, b, c 同上
order by a DESC, b DESC, c DESC 同上
where a =const order by b, c Y，where 使用索引的最左前缀定义为常量，则 order by 能使用索引
where a =const and b=const order by c 同上
where a =const and b>const order by b, c 同上
order by a ASC, b DESC, c DESC N，排序不一致
where g =const order by b, c N，丢失 a 索引
where a =const order by c N，丢失 b 索引
where a =const order d N，d 不是索引的一部分
where a in (……) order by b, c N，对于排序来说，多个相等条件也是范围查询，索引失效

group by 关键字分组优化

  • group by 实质是先排序后进行分组，也遵循和 order by 一样的原则

  • 当无法使用索引列，增大 max_length_for_sort_data 参数的设置+增大 sort_buffer_size 参数的设置

  • where 高于 having，能写在 where 限定的条件就不要去 having 限定了

慢查询日志

概念

MySQL 的慢查询日志是 MySQL 提供的一种日志记录，它用来记录在 MySQL 中响应时间超过阈值的语句，具体指运行时间超过 long_query_time 值的 SQL，则会被记录到慢查询日志中，然后可以结合之前 explain 进行全面分析。

long_query_time 的默认值为 10，意思是运行 10 秒以上的 SQL 语句。

使用方法

  • 查看是否开启慢查询日志以及日志的存放位置：SHOW VARIABLES LIKE ‘%slow_query_log%’

  • 开启慢查询日志：set global slow_query_log = 1

  • 查看默认慢查询的阈值：SHOW VARIABLES LIKE ‘long_query_time%’;

  • 设置慢查询的阈值：set global long_query_time=3;，设置完毕之后需要重新开启一个 MySQL 连接，才能看到 long_query_time 发生的变化

  • 查看当前系统有多少条慢查询日志：show global status like ‘%slow_queries%’

上述命令只在当前 MySQL 服务中生效，如果 MySQL 服务重启，将会失效。如果想要永久生效，需要在数据库配置文件 my.cnf 中进行配置。

日志分析工具

在生产环境中，如果要手工分析日志，手工查找、分析 SQL，显然是个体力活，MySQL 提供了日志分析工具 mysqldumpslow。

常用：

  • 得到返回记录集最多的 10 个 SQL
    mysqldumpslow -s r-t 10 /var/lib/mysql/主机名-slow.log
  • 得到访问次数最多的 10 个 SQL
    mysqldumpslow -s c-t 10 /var/lib/mysql/主机名-slow.log
  • 得到按照时间排序的前 10 条里面含有左连接的查询语句
    mysqldumpslow -s t-t 10 -g “left join” /var/lib/mysql/主机名-slow.log
  • 另外建议在使用这些命令时结合 | 和 more 使用，否则有可能出现爆屏情况
    mysqldumpslow -s r-t 10 /var/lib/mysql/主机名-slow.log | more

show profiles

show profiles 是 MySQL 提供可以用来分析当前会话中语句执行的资源消耗情况，可以用于 SQL 的调优测量，比 explain 更牛逼！！默认情况下，参数处于关闭状态，并保存最近 15 次的运行结果。

  • 查看是否开启此工具：show variables like ‘profiling%’

  • 查看结果：show profiles;

  • 诊断 SQL，show profile cpu,block io for query 上一步前面的问题 SQL 数字号码；

日常开发中，如果诊断 SQL 出现了下面几条，则表示很危险！！

  • converting HEAP to MyISAM：查询结果太大，内存都不够用了往磁盘上搬了
  • Creating tmp table：拷贝数据到临时表，创建临时表用完再删除
  • Copying to tmp table on disk：把内存中临时表复制到磁盘，危险！！
  • locked

全局日志查询

开启全局日志查询：set global general_log=1; set global log_output=‘Table’;

如果开启了全局日志查询，你所执行的所有 SQL 语句都会记录到 MySQL 库里的 general_log 表，可以用下面的命令查看 select * from mysql.general_log; ，永远不要在生产环境开启这个功能。
