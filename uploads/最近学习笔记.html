为什么 mysql 一定要有主键，并且推荐使用整型的自增主键？

mysql 的 innodb 存储引擎默认会为主键创建聚簇索引，如果该表没有设置主键，那么 mysql 会为该表生成一个伪主键，然后基于该伪主键创建 b+ tree 索引。
b+ tree 是多路平衡搜索树，在存储数据时，同时会对数据进行排序，如果索引列是自增的整型，排序的成本低，至少比 UUID 等字符串低太多。

jvm 调优的目的

减少 full gc 出现的次数，减少用户线程停顿的时间 (stop the word)，避免出现 OOM 异常。
minor gc 也会 stop the world，但是时间非常短，可以忽略不计，full gc 需要对整个堆进行扫描，stop the world 的时间非常长，影响用户体验。
如果老年代的对象大多数都是些垃圾对象，那么为什么不将这些垃圾对象在新生代就回收掉呢？从而可以避免 full gc。
注意动态年龄判定，它会导致要回收的部分对象直接进入到老年代，从而不会被 minor gc。

单例模式的应用场景

单例模式指一个类只有一个实例，然后对外提供公开的访问方法。
在有些程序里，我们有一些很重要的实例不能实例化多次，比如线程池、数据库连接池、工厂对象。

懒汉式单例模式

懒汉：在真正使用实例的时候才创建该实例。JDK 很多底层是通过双端检验锁机制实现单例模式的。
单例模式为什么需要 volatile 修饰实例变量？
new 一个对象的过程，指令重排序，未完成初始化而导致空指针异常。
volatile 禁止指令重新排序，内存栅栏保护屏障内的指令不被优化重排序。

饿汉式单例模式

饿汉：在加载类的时候就创建了该类的唯一实例对象。

饿汉式单例模式天生线程安全，为什么？
首先要了解类加载的过程。

  1. 加载：将该类的字节码文件从磁盘加载到内存，将这个类定义的静态存储结构转换为在方法区的存储结构，最后生成一个 Class 对象，作为在方法区访问各个数据的入口。
  2. 验证：验证字节码文件的内容是否符合 JVM 的规范
  3. 准备：为该类的静态成员默认初始化
  4. 解析：将该类的符号引用替换为直接引用
  5. 初始化：为该类的静态成员显示初始化。

在多线程的环境下，即使有多个线程请求去加载这个类，但是最终只会有一个类去加载该类，该实现通过 loadClass() 方法内的 synchronized 保证。当一个类已经被类加载器加载过，JVM 不会再重复加载该类，所以类的静态成员变量和静态代码块才只会执行一次。

静态内部类单例模式

对饿汉式单例模式进行改进，将类的加载通过内部类的方式进行延迟。
静态内部类，也是利用了类的加载机制，可以保证线程安全。

破坏单例模式

  1. 反射攻击

    通过反射的方法，取得私有构造方法的权限，然后通过构造方法创建实例对象。

    new Instance() 实际上就是通过构造方法创建实例对象，不过如果 SingleInstance 如果是 枚举类型，那么是不允许通过 new Instance() 创建对象的，所以枚举的方式实现单例模式，天然支持反射攻击。

  2. 对象序列化

为什么现在的计算机是 cpu 多级缓存架构，为什么要加缓存？

当下计算机 cpu 的运算速度远远大于内存，为了提高 cpu 和内存通信的时效性，加入了三级 cpu 缓存，cpu 缓存的速度大于内存。

java 内存模型 (jmm) 正是基于 cpu 多级缓存架构建立的，目的也是为了保证时效性，屏蔽操作系统与底层硬件的差异。

jmm 缓存数据不一致的解决方法

  1. 总线加锁 (效率差)，对消息总线进行加锁，假设某线程正在读写主内存共享数据，其他也想读写此共享变量的线程会被阻塞，当此线程执行完毕，其他线程才能继续……，相当于将多线程串行执行了，当前效率低咯。
  2. MESI 缓存一致性协议，
    多核 cpu 首先从主内存读取共享数据到自己的高速缓存，当某 cpu 操作完此数据，该数据会被立马同步到主内存，其他 cpu 通过总线嗅探机制可以感知数据已经被修改，从而使自己缓存内的数据失效，然后重新从主内存中读取。

volatile 的底层原理

volatile 是 java 提供的一个关键字，它是轻量级的线程同步机制，被 volatile 修饰的变量会被 java 编译器编译成带 lock 前缀的汇编指令，lock 前缀指令有 3 个作用。如果是没有被 volatile 修饰的变量，那么编译的汇编指令是不带 lock 前缀的，所以也不会触发总线嗅探机制。

  1. 保证各线程会将操作完的数据立马写回到主内存，注意是立马！
  2. 写回主内存的过程中，数据会经过消息总线，由于 MESI 缓存一致性协议，其他线程会通过总线嗅探机制，发现此共享变量被修改了，从而导致自己工作内存的数据失效，需要重新从主内存读取最新的值
  3. lock 前缀的汇编指令会在数据写回到主内存时加锁，也就是在 store 前使用 lock 原子指令，为什么呢？因为 cpu 总线嗅探机制，其他线程已经发现了该数据已经被修改了，所以它们从主内存中读取最新的值，而写回主内存和从主内存中读，这中间存在一个时间差，为了保证其他线程从主内存中读取的是最新的值，所以需要在写回的时候加锁，也就是 lock 原子操作，只要在我写回之后你才去读。

为什么是立马同步回主内存？

因为当前线程修改完此共享数据之后，假如它下面还有很多业务代码，为了保证多线程通信整体的时效性，所以是立马同步回主内存，而不是由操作系统决定在某个不确定的时间去同步。

为什么 volatile 不保证原子性？

volatile 只保证单个操作的原子性，也就是那 8 个原子操作，并不保证多个操作的原子性。比如 i=1，简单的赋值，是原子的。i=i+1，首先读取 i 的值，然后执行+1 操作，然后写回到 i，不是原子的。这也是 volatile 的适用场景，如果只是对变量进行访问或者赋值，没问题；但是是对变量进行其他操作，会导致数据不一致。

假设有两个线程 A 和 B 都对共享变量操作完毕了，要写回主内存，其中呢，A 线程写回主内存的时候，执行 lock 原子操作保证 B 线程无法写入，A 写入时 B 通过总线嗅探机制，发现该内存地址的值被修改了，所以 B 的数据失效，需要从主内存拷贝该数据的最新值，但是 B 之前执行的操作就被覆盖了，这就是数据的不一致，要想保证原子性，还是得加锁或者 CAS 原子操作。

并发存在的问题？

在多线程的环境下，多个线程同时访问或操作某一共享数据，如果不进行同步控制，可能造成数据不一致和死锁。
怎么进行同步控制？
将业务代码进行封装，然后加锁 lock 或者 synchronized，保证该业务执行时的原子性。

AQS 底层实现原理

AQS 即抽象的队列同步器，是 Java 提供的一套灵活的线程同步机制，像 ReentrantLock、CountDownLatch、CyclicBarrier 的底层都是基于 AQS 实现的，它可以保证在多线程环境下共享数据的一致性。例如 AQS 可以保证在购物系统中，如果有多个用户同时下单，在库存不足的情况下不会出现超卖的现象。

AQS 的三板斧:自旋，LockSupport，CAS

属性：

  1. state：当前锁的状态，加锁的次数，state 相当于互斥资源，通过 CAS 保证多个线程最终只能有一个线程能够拿到它。

  2. lockHolder：锁的持有者

  3. waiterQueue：等待队列，记录获取锁失败的线程

  4. unsafe：此处的作用是基于 CAS 原子操作安全的修改 state 状态

    Unsafe 类是一个魔法类，它可以绕过 jvm，直接操作内存的数据，因为 Unsafe 类大多数都是基于 C++实现的 native 方法

    注意：不能通过 new 的方式直接创建 Unsafe 类的实例，Unsafe 类的构造方法是 private 修饰的，所以只能通过反射的方式取得 Unsafe 类的实例。Unsafe 类中有一个静态的常量 unsafe 实例（Unsafe 类的成员），通过反射取出该属性

  5. stateOffset：CAS 原子操作修改 state 时，需要知道 state 在内存的起始地址和偏移地址

方法：

  1. void compareAndSwapState()：封装 Unsafe 类下的 CAS 方法，修改 state 属性

    Unsafe 的 compareAndSwap() 的四个参数: 此对象 (其实就是取该对象在内存的起始地址)、偏移量、期望值、更新值。

  2. boolean acquire()：当前线程尝试去获取锁，返回是否获取成功

  3. void lock()：加锁

  4. void unlock()：解锁

知识点：

为什么自旋 for 循环不用 yield()？
yield() 的作用是让当前线程让出 CPU 的执行权，但是不代表当前线程不使用 CPU，所以此时的 for 循环仍然会执行，当前线程仍然会消耗 CPU。

为什么自旋 for 循环不用 sleep(long)？
无法确定到底应该 sleep(long) 多长时间，如果某业务的执行时间很短，我 sleep() 的时间又很长，那样 cpu 会出现空闲时间，降低 cpu 的利用率；如果某业务的执行时间很长，我 sleep() 的时间又很短，那么系统会进行多次线程的上下文切换，增大开销。

庆幸的是 Unsafe 类提供给了我们 park(thread) 和 unpark(thread) 方法，park(thread) 可以阻塞当前线程，直到被 unpark(thread)。实际使用时不建议直接使用 Unsafe 类，因为它被 java 认为是一个不安全的类（从类名也可以知道），建议使用 LockSupport，它对 park() 进行的封装。

ConcurrentLinkedQueue 底层是基于 CAS 实现的，而不像其他 Queue 本身就是基于 AQS 实现的。

wait() 和 notify() 是阻塞与随机唤醒，要实现公平锁的话还得是 park(thread) 和 unpark(thread)。
