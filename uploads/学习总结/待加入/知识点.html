Java 基础

面向过程和面向对象的区别

本质区别：面向过程是以执行一件事情的过程为核心的编程思想，面向对象是以这件事情所涉及的对象为核心的编程思想。

  • 面向过程：面向过程的性能一般比面向对象高。.c—> 二进制机器码。

  • 面向对象：面向对象易维护、易复用、易扩展，面向对象有封装、继承、多态三大特性，所以可以设计出低耦合的系统。.java—>.class—> 二进制机器码。Java 是半编译半解释语言，首先.java 文件被编译器编译为.class 文件，然后.class 文件被解释器逐行解释为二进制机器码。

谈谈你对 Java 面向对象的理解

在我理解，面向对象是一种编程思想，它是对现实世界模型的自然延伸，现实世界中的任何物体都可以归为一类事物，而每一个个体都是该类事物的实例，面向对象的编程就是以对象为中心，以消息为驱动，所以程序=对象+消息。

面向对象的三大特性：

  • 封装：将一类事物的属性和行为抽象为一个类，使其属性私有化，行为公开化，这样做使得代码的复用性更高
  • 继承：将一类事物共有的属性和行为抽象为一个父类，而它的子类既有父类的属性和行为，也有自己特定的属性和行为，这样做进一步提升了代码的复用性
  • 多态： 允许父类引用（或接口）指向子类（或实现类）对象， 减少程序的耦合度

Java 八大基本数据类型

  • byte(8)、short(16)、int(32)、long(64)
  • float(32)、double(64)
  • char(16)
  • boolean(~)： boolean 只有两个值：true、false，可以使用 1 bit 来存储，但是具体大小没有明确规定。JVM 会在编译时期将 boolean 类型的数据转换为 int，使用 1 来表示 true，0 表示 false。JVM 支持 boolean 数组，但是是通过读写 byte 数组来实现的。

深入剖析 Java 中的装箱和拆箱

Integer 的缓存池

在 jdk1.8 所有的数值类缓冲池中，Integer 的缓冲池 IntegerCache 很特殊，这个缓冲池的下界是 -128，上界默认是 127，但是这个上界是可调的，在启动 jvm 的时候，通过 -XX:AutoBoxCacheMax=来指定这个缓冲池的大小，该选项在 JVM 初始化的时候会设定一个名为 java.lang.IntegerCache.high 系统属性，然后 IntegerCache 初始化的时候就会读取该系统属性来决定上界。

char 型变量能不能存储一个中文汉字

char 类型可以存储一个中文汉字，因为对 char 类型的字符 Java 使用的编码方式是 Unicode，一个 char 类型占 2 个字节，而一个中文字符也是两个字节，所以放一个中文是没问题的。

补充：使用 Unicode 意味着字符在 JVM 内部和外部有不同的表现形式，在 JVM 内部都是 Unicode，当这个字符被 JVM 从内部转移到外部时（例如存入文件系统中），需要进行编码转换。所以 Java 中有字节流和字符流，以及在字符流和字节流之间进行转换的转换流，如 InputStreamReader 和 OutputStreamReader，这两个类是字节流和字符流之间的适配器类，承担了编码转换的任务。

String 类型为什么不可变

String 类的底层使用 private final 类型的数组存储数据，数组一旦初始化之后就不能再引用其他数组，子类也无法继承 String 类修改数组内部元素，并且 String 类本身也没有提供修改数组内部元素的方法，所以可以保证 String 不可变。

String 类型不可变的好处

1. 可以缓存 hash 值

String 类中有一个 hash 字段，它会缓存计算的 hash 值，因为 String 类型不可变，所以该 hash 值只需要计算一次，提高了效率，比如将 String 用做 HashMap 的 key。

2. String Pool 的需要

如果一个 String 对象已经被创建过了，那么就会从 String Pool 中取得引用。只有 String 是不可变的，才可能使用 String Pool，否则引用的 String 值不确定。

3. 线程安全

String 不可变性天生具备线程安全，可以在多个线程中安全地使用。

Object 类通用方法

重写 equals() 方法

  1. 检查是否为同一个对象的引用，如果是直接返回 true；
  2. 检查是否是同一个类型，如果不是，直接返回 false；
  3. 将 Object 对象进行转型；
  4. 判断每个关键域是否相等。

hashCode() 方法

hashCode() 返回对象的哈希值，而 equals() 是用来判断两个对象是否等价。等价的两个对象 hashCode 值一定相同，但是 hashCode 值相同的两个对象不一定等价，这是因为计算哈希值具有随机性，两个值不同的对象可能计算出相同的哈希值。

在覆盖 equals() 方法时应当总是覆盖 hashCode() 方法，保证等价的两个对象哈希值也相等。

HashSet 和 HashMap 等集合类使用了 hashCode() 方法来计算对象应该存储的位置，因此要将对象添加到这些集合类中，需要让对应的类实现 hashCode() 方法。

clone() 方法

  • 重写 clone() 方法： 如果一个类没有实现 Cloneable 接口又调用了 clone() 方法，就会抛出 CloneNotSupportedException。
  • 浅拷贝： 拷贝对象和原始对象的引用类型引用同一个对象。
  • 深拷贝： 拷贝对象和原始对象的引用类型引用不同对象。
  • 深拷贝的替代方法： 使用 clone() 方法来拷贝一个对象即复杂又有风险，它会抛出异常，并且还需要类型转换，Effective Java 书上讲到，最好不要去使用 clone()，可以使用拷贝构造函数或者拷贝工厂来拷贝一个对象。

static 关键字

静态内部类与非静态内部类之间存在一个最大的区别：非静态内部类在编译完成之后会隐含地保存着一个引用，该引用是指向创建它的外围类，但是静态内部类却没有。没有这个引用就意味着：1. 它的创建是不需要依赖外围类的创建。2. 它不能使用任何外围类的非 static 成员变量和方法。

非静态代码块与构造函数的区别是： 非静态代码块是给所有对象进行统一初始化，而构造函数是给对应的对象初始化，因为构造函数是可以多个的，运行哪个构造函数就会建立什么样的对象，但无论建立哪个对象，都会先执行相同的构造代码块。也就是说，构造代码块中定义的是不同对象共性的初始化内容。

抽象类与接口的区别

  1. 抽象类是对概念的抽象，比如美国人、中国人、日本人可以抽象为人类，接口是对行为的扩展，比如中国人会功夫，而美国人、日本人不会功夫，所以中国人可以实现功夫接口
  2. 一个类只能继承一个抽象类，但是可以实现多个接口
  3. 接口的成员必须是 public 修饰，抽象类的成员没有此限制
  4. 接口的成员属性必须是 public final static，抽象类的成员属性没有此限制
  5. 接口只有抽象方法，抽象类既可以有抽象方法也可以有非抽象方法

重载和重写

1. 重载（Overload）

重载：编译时多态，指同一个类中有多个同名方法，它们的参数列表不一样，所以可以根据不同的传参有不同的逻辑处理。

2. 重写（Override）

重写：运行时多态，指在子父类继承体系中，子类存在与父类同名的方法，该方法是对父类方法的改造。在使用时，将父类的引用指向子类的实例，根据指向子类的不同，执行不同子类下重写的方法。

为了满足里式替换原则，重写有以下三个限制：

  • 子类方法的访问权限必须大于等于父类方法；
  • 子类方法的返回类型必须是父类方法返回类型或为其子类型。
  • 子类方法抛出的异常类型必须是父类抛出异常类型或为其子类型。

使用 @Override 注解，可以让编译器帮忙检查是否满足上面的三个限制条件。

继承体系中成员初始化的顺序

  • 父类（静态变量、静态语句块）
  • 子类（静态变量、静态语句块）
  • 父类（实例变量、普通语句块）
  • 父类（构造函数）
  • 子类（实例变量、普通语句块）
  • 子类（构造函数）

继承体系中方法执行的顺序

在调用一个方法时，先从本类中查找看是否有对应的方法，如果没有再到父类中查看，看是否从父类继承来。否则就要对参数进行转型，转成父类之后看是否有对应的方法。总的来说，方法调用的优先级为：

  • this.func(this)
  • super.func(this)
  • this.func(super)
  • super.func(super)

finally 块不被执行的四种情况

  1. 在 finally 语句块第一行发生了异常；
  2. 在异常发生前使用 System.exit(int) 退出程序；
  3. 程序所在的线程死亡；
  4. 关闭 CPU。

throw 和 throws 的区别

throw：

  • throw 语句用在方法体内，表示抛出异常，由方法体内的语句处理；
  • throw 是具体向外抛出异常的动作，所以它抛出的是一个异常实例，执行 throw 一定是抛出了某种异常。

throws

什么是 Java 序列化

序列化就是一种用来处理对象流的机制，也就是将对象进行流化。可以对流化后的对象进行读写操作，也可以将流化后的对象传输于网络之间。序列化是为了解决在对对象流进行读写操作时所引发的问题。

序列化的实现：将需要被序列化的类实现 Serializable 接口，该接口没有需要实现的方法，Serializable 只是为了标注该对象是可被序列化的，然后使用一个输出流（如：FileOutputStream）来构造一个 ObjectOutputStream（对象流）对象，接着，使用 ObjectOutputStream 对象的 writeObject（Object obj）方法就可以将参数为 obj 的对象写出（即保存其状态），要恢复的话则用输入流。

Comparable 和 Comparator 的区别

Comparable：自然排序，待排序的类实现 Comparable 接口，重写 compareTo(Object obj) 方法

Comparator：比较器排序，通过实现 Comparator 接口开发自定义排序比较器，重写 compare(Object obj1, Object obj2) 方法

  • 相同点：都可以自定义排序规则实现排序，返回值的类型都为 int 类型
  • 不同点：Comparator 的扩展性更强，可以根据需求开发多个自定义排序比较器，也就可以实现多种排序方式。而一个类只能实现一次 Comparable 接口，所以只可以实现一种排序方法。

HashSet 如何检查重复

当你把对象加入HashSet时，HashSet 会先计算对象的hashcode值来判断对象加入的位置，同时也会与其他加入的对象的 hashcode 值作比较，如果没有相符的 hashcode，HashSet 会假设对象没有重复出现。但是如果发现有相同 hashcode 值的对象，这时会调用equals()方法来检查 hashcode 相等的对象是否真的相同。如果两者相同，HashSet 就不会让加入操作成功。

HashMap 的底层数据结构

HashMap 的底层数据结构为数组+链表+红黑树 (jdk1.8)，理想情况下 HashMap 查找元素的时间复杂度度为 O(1)，实际平均时间复杂度为 O(N/M)，N 表示键值的数量，M 表示数组的长度。HashMap 基于数组可以以 O(1) 的时间复杂度计算出键值在数组的哪个位置；基于链表解决 hash 冲突，但是在链表上查询数据的时间复杂度为链表的长度 O(N)，所以在 jdk1.8 对 HashMap 进行了优化，引用了红黑树，在红黑树上查询数据的时间复杂度为红黑树的高度 O(logN)。

红黑树的 5 大性质

  1. 红黑树的节点有两种颜色：红色和黑色
  2. 根节点必须是黑色
  3. 叶子节点必须是黑色，且为 NULL
  4. 红色节点的左右子节点必须是黑色
  5. 任意节点到其所有的叶子节点的路径，出现的黑色节点一样多

HashMap 中 put() 过程

  1. 首次 put 元素时，对数组进行初始化，默认数组的大小为 16，也可以自定义大小，但必须是 2 的指数次幂，如果不是，默认会转为 2 的指数次幂
  2. 计算 key 的 hash 值
  3. 计算此键值在数组的位置
  4. 如果此位置为空，则直接构造一个节点，将该节点放在此位置
  5. 否则循环该位置的链表，比较链表上节点的 key 和 put 键值的 key，如果相等则更新该节点；如果不相等则插入新节点

jdk1.8 中的 HashMap 对 jdk1.7 进行了优化，第 5 步是不一样的，jdk1.7 是以头插法插入新节点，而 jdk1.8 是尾插法。

为什么 HashMap 初始化时容量为 2 的指数次幂，并且每次扩容都是 2 倍

HashMap 在 put 元素时，需要计算将该键值放在数组哪个位置，为了不超出数组的范围，比较直观的方式是 hash%length，但是 HashMap 实际使用的方式是 hash&(length-1)，可以达到与 hash%length 一样的效果，length 表示数组的长度。使用位运算&代替了％取余元素，能够大大提高计算效率，但是使用此方法的前提是 length 为 2 的指数次幂。

当往 HashMap 放入 100w 个元素时，每次都要计算该键值应该放在数组中的哪个位置，同时 HashMap 扩容时需要 rehash，重新计算键值应该放在新数组哪个位置，所以计算的次数远远大于 100w 次，使用位运算极大的提高了计算效率。

HashMap 的加载因子为什么是 0.75

加载因子的作用是计算 HashMap 扩容时的阈值，HashMap 存储键值对的数量大于阈值时，HashMap 执行扩容 resize()。

极端一点，取加载因子为 1，也就是 HashMap 存储的键值数量大于数组的长度才执行扩容。HashMap 通过 hash 值计算键值放在数组哪个位置的，具有随机性，避免不了 hash 碰撞问题，如果 HashMap 需要使存储的键值数量大于数组的长度时才执行扩容，必然会导致更多的 hash 碰撞，那么数组位置上的链表或者红黑树会变得十分复杂，从而导致 HashMap 查找数据的时间复杂度过高。

当加载因子为 0.5 时，表示最多只使用了数组的一半空间就执行扩容，虽然一定程度减少了 hash 碰撞，但是会造成空间浪费，每次只使用了实际的一半内存。
所以为了同时节省时间和空间，java 为 HashMap 取了中间值 0.75，c# 是 0.72，实际上有一种通过概率的方式可以计算出加载因子最优为 0.693。

jdk1.8 HashMap 中链表什么时候转为红黑树

当往数组的某位置 put 一个新的元素，该位置链表的长度大于等于 8，并且数组的长度小于 64，HashMap 首先会去尝试扩容 resize()，而不是直接转红黑树；如果数组的长度大于等于 64，则转为红黑树。

还有一点需要注意：是当链表的长度达到 8 时，并且此时又想 put 一个新的键值在该位置上，会将新的节点插入到链表的尾部，然后才转为红黑树，所以真正转红黑树时链表的长度是 9。

什么是快速失败(fail-fast)？

快速失败 (fail-fast) 是 Java 集合的一种错误检测机制。在使用迭代器对集合进行遍历的时候，我们在多线程下操作非安全失败 (fail-safe) 的集合类可能就会触发 fail-fast 机制，导致抛出异常。另外，在单线程下，如果在遍历过程中对集合对象的内容进行了修改的话也会触发 fail-fast 机制。

注：增强 for 循环也是借助迭代器进行遍历。

使用 modCount 属性记录 ArrayList 结构发生变化的次数。结构发生变化是指添加或者删除元素的操作，或者是调整内部数组的大小，仅仅只是设置元素的值不算结构发生变化。如果我们在使用迭代器遍历集合时，对其结构进行修改的话，就会改变 modCount 的值，进而导致 modCount != expectedModCount ，从而抛出 ConcurrentModificationException 异常。

而通过 Iterator 的方法修改集合的话会修改 expectedModCount 的值，所以不会抛出异常。

总结：在使用迭代器遍历的时候，禁止使用 list.remove() 或者 list.add() 此类型的操作，如果非要遍历的时候添加或者删除元素，可以使用 iterator.hasNext() 和 iterator.next() 的方式。

java.util包下面的所有的集合类都是fail-fast的，而java.util.concurrent包下面的所有的类都是fail-safe的。

什么是安全失败(fail-safe)？

采用安全失败机制的集合容器，在遍历时不是直接在集合内容上访问的，而是先复制原有集合内容，在拷贝的集合上进行遍历。所以，在遍历过程中对原集合所作的修改并不能被迭代器检测到，故不会抛 ConcurrentModificationException 异常。

说说 synchronized 关键字和 volatile 关键字的区别

volatile 关键字主要用于解决变量在多个线程之间的可见性，而 synchronized 关键字解决的是多个线程之间访问资源的同步性。

synchronized 和 ReentrantLock 的区别

  1. synchronized 是 java 的关键字，属于 jvm 层面；ReentrantLock 是 java 的具体类，属于 jdk 层面
  2. synchronized 只支持非公平锁；ReentrantLock 支持公平与非公平锁
  3. synchronized 由 jvm 自动释放锁；ReentrantLock 需要手动释放锁，否则可能发生死锁
  4. synchronized 等待的线程不可中断，被 synchronized 阻塞的线程将一直处于等待状态；ReentrantLock 等待可中断
  5. synchronized 不可以绑定 Condition；ReentrantLock 可以绑定 Condition，实现精准唤醒

中断线程

1. InterruptedException

通过调用一个线程的 interrupt() 来中断该线程，如果该线程处于阻塞、限期等待或者无限期等待状态，那么就会抛出 InterruptedException，从而提前结束该线程。但是不能中断 I/O 阻塞和 synchronized 锁阻塞。（ReentrantLock 锁阻塞可中断）

2. interrupted()

如果一个线程的 run() 方法执行一个无限循环，并且没有执行 sleep() 等会抛出 InterruptedException 的操作，那么调用线程的 interrupt() 方法就无法使线程提前结束。

但是调用 interrupt() 方法会设置线程的中断标记，此时调用 interrupted() 方法会返回 true。因此可以在循环体中使用 interrupted() 方法来判断线程是否处于中断状态，从而提前结束线程。

3. Executor 的中断操作

调用 Executor 的 shutdown() 方法会等待线程都执行完毕之后再关闭，但是如果调用的是 shutdownNow() 方法，则相当于调用每个线程的 interrupt() 方法。

对 volatile 的理解

volatile 是 Java 的关键字，是轻量级的线程同步机制，保证了可见性、不保证原子性、禁止指令重排。

volatile 禁止指令重排的原理是什么

Java 代码在执行前会被编译成 Java 指令，为了提高执行效率和性能，这些指令可能会被优化重新排序，在单线程的环境下，指令优化重排不会影响程序的最终执行结果；但是在多线程的环境下，程序最终的执行结果不确定，所以需要禁止该变量指令重排。

JVM 会为被 valotile 修饰的变量或者代码块加入内存屏障（内存栅栏），内存屏障是一个 CPU 指令，它可以保证重排序时后面的指令不会放到内存屏障之前。

在什么地方使用过 volatile

双端校验机制的单例设计模型并不能完全保证单例，需要使用 volatile 关键字修饰共享实例。

什么是 CAS

CAS 表示比较并交换，它是一种不需要加锁就可以保证线程安全的一种方式，比较并交换指的是某线程对共享变量进行修改后，将修改值写入到主内存，首先会设置一个期望值，将此期望值和主内存的实际值进行比较，如果相同，则写入；如果不相同，则使用新值再执行修改操作，直到期望值和实际值一致。

CAS 在 Java 中通过 java.util.concurrent.atomic 包下的类实现，该包下的类底层通过操作 UnSafe 类下的 native 方法，CAS 操作被设计成一条 CPU 并发原语，属于操作系统级别，在执行 CAS 原语时，保证原子性，不会被中断，不会造成数据一致性问题。

AtomicInteger、AtomicReference。

CAS 的弊端

  1. 如果循环时间过长，系统开销较大
  2. 只能保证单个变量的原子操作
  3. ABA 问题

什么是 ABA 问题，怎么解决

ABA 问题：比如线程 1 从内存位置 V 取出 A，线程 2 同时也从内存取出 A，并且线程 2 进行一些操作将值改为 B，然后线程 2 又将 V 位置数据改回 A，这时候线程 1 进行 CAS 操作发现内存中的值依然是 A，然后线程 1 操作成功。尽管线程 1 的 CAS 操作成功，但是不代表这个过程没有问题。

使用版本号机制解决 ABA 问题：AtomicStampedReference。

CopyOnWrite 容器的原理

CopyOnWrite 容器可以保证在多线程的环境下，读写数据不互斥。当一个线程往容器添加数据时，不是在原容器添加，而是复制出一个新的容器，往新容器添加数据，添加完成后将原容器的引用指向新容器，即当前线程在添加数据时，不影响其他线程读数据。但是写写仍然是互斥的，添加数据的操作使用 ReentrantLock 进行加锁。

CopyOnWrite 容器的缺点：写入的数据无法保证实时更新，需要延迟才能读取。

线程池的 4 种拒绝策略

  • 默认的拒绝策略 AbortPolicy

    直接抛出 RejectedExecutionException 异常，阻止系统正常运行

  • 回退到调用者线程 CallerRunsPolicy

    该策略既不会抛弃任务，也不会抛出异常，而是将某些任务回退到调用者线程，调用者线程可能再次提交该任务，也有可能去扩展任务队列的容量。显然这样做不会真的丢弃任务，但是，任务提交线程的性能极有可能会急剧下降

  • 抛弃任务队列中等待最久的任务 DiscardOldestPolicy

    抛弃队列中等待最久的任务，然后把当前任务加入队列中尝试再次提交当前任务

  • 抛弃当前任务 DiscardPolicy

    直接丢弃任务，不予任何处理也不抛异常。如果允许任务丢失，这是最好的一种方案

内存在堆中的分配策略

  • 对象优先在 Eden 区进行分配
  • 大的数组和字符串直接进入老年代
  • 当 survivor 区对象的年龄大于设置的阈值时，对象会晋升到老年代
  • 动态对象年龄判定：如果 survivor 区有大量相同的年龄的对象，并且相同年龄的这些对象所占的内存空间大于 survivor 空间的一半，那么 jvm 会将大于等于该年龄的对象晋升到老年代。
  • 空间分配担保：每一次 minor gc 都会进行空间分配担保。判断老年代最大可用的连续空间是否大于新生代所有对象的空间总和，如果大于的话，那么直接执行 minor gc；如果不大于的话，那么需要再检查 JVM 是否允许空间分配担保，如果允许的话，继续判断老年代最大可用的连续空间是否大于历次出现此种情况时的平均大小，如果大于的话，尝试执行一次 minor gc；如果不大于的话，表示空间分配担保失败，执行 full gc

Minor GC 的过程

  1. 从 eden、survivor 0 复制仍存活对象到 survivor 1，如果有对象的年龄达到了晋升到老年代的标准，那么这些对象会复制到老年代，又如果 survivor 1 区内存不足，那么这些对象也会复制到老年代，否则复制到 survivor 1 区，同时对象的年龄加 1
  2. 清空 eden、survivor 0
  3. survivor 0 和 survivor 1 互换

HotSpot 为什么要分为新生代和老年代？

双亲委派机制

双亲委派模型保证了 Java 程序的稳定运行，可以避免类的重复加载（JVM 区分不同类的方式不仅仅根据类名，相同的类文件被不同的类加载器加载产生的是两个不同的类），也保证了 Java 的核心 API 不被篡改。

如果没有使用双亲委派模型，而是每个类加载器加载自己的话就会出现一些问题，比如我们编写一个称为 java.lang.Object 类的话，那么程序运行的时候，系统就会出现多个不同的 Object 类。

TCP 协议如何保证可靠传输

  • 校验和机制：比较校验和，检验数据报在传输的过程中是否出现丢失，常用的算法有循环冗余校验算法
  • 确认机制（ARQ 协议）：对数据报进行有序的确认，需要先对数据报进行编号
    • 停止等待 ARQ 协议
    • 连续 ARQ 协议
  • 超时重传机制：发送方每发送一个数据报，会同步启动一个计时器，在计时器指定的时间内，如果没有收到这个数据报的确认信息，发送方会重传此数据报
  • 流量控制：通过滑动窗口算法控制发送方的发送速率，使接收方来得及处理数据报
  • 拥塞控制：降低整个网络的拥塞程度
    • 慢开始
    • 拥塞避免
    • 快重传与快恢复

从输入 URL 到页面加载到底发生了什么？

  1. 对域名进行解析，返回 IP 地址（DNS 查找过程：浏览器缓存、本地缓存、DNS 服务器缓存）
  2. 浏览器封装 HTTP 数据报，向 web 服务器发送一个 HTTP 请求
  3. 服务器处理请求
  4. 服务器返回响应结果
  5. 浏览器解析响应结果并渲染页面
  6. 关闭连接

Cookie的作用是什么？和Session有什么区别？

Cookie 和 Session 都是用来跟踪浏览器用户身份的一种会话方式。

  • Cookie 数据保存在客户端，Session 数据保存在服务器端
  • 相对来说 Session 的安全性更高，如果要在 Cookie 中存储一些敏感信息，不要直接写入 Cookie 中，最好能将 Cookie 信息进行加密，然后再去服务器端解密。

在服务端保存 Session 的方法很多，最常用的就是内存和数据库。既然 Session 存放在服务器端，那么我们如何实现 Session 跟踪呢？大部分情况下，我们都是通过在 Cookie 中附加一个 Session ID 来方式来跟踪。如果 Cookie 被禁用，最常用的就是利用 URL 重写技术，把 Session ID 直接附加在 URL 路径的后面。

HTTPS 原理

要解决的问题：

  1. HTTP 使用明文进行通信，数据可能被窃取
  2. HTTP 未验证通信方的身份，通信方可能被伪装
  3. HTTP 不保证数据的完整性，数据可能被篡改

怎么解决？

  1. 基于混合加密算法（对称加密算法+非对称加密算法）对数据加密
  2. 数据签名绑定证书，验证通信方的身份
  3. 单向 Hash 加密算法保证数据的完整性

过程：

  1. 客户端希望与服务器通信，故想得到服务器的公钥
  2. 服务器需要向第三方数字证书认证机构申请证书，申请信息+公钥 ——> 单向 Hash 算法生成摘要——> 使用机构的私钥对摘要加密生成数据签名——> 将数字签名+申请信息+公钥绑定生成证书——> 将证书发送给客户端
  3. 客户端得到服务器发送过来的证书，首先要验证该数字证书认证机构的合法性，浏览器内置一些常见的第三方数字证书认证机构，然后验证该证书的合法性，申请信息+公钥——> 使用同样的单向 Hash 算法生成摘要 1——> 使用该机构的公钥对数字签名进行解密生成摘要 2——> 将摘要 1 与摘要 2 进行比较——> 验证通过之后，即可取出服务器的公钥
  4. 客户端生成对称加密算法的密钥，使用服务器的公钥加密该密钥，将加密的报文发送给服务器
  5. 服务器使用自己的私钥解密该报文，取出对称加密算法的密钥，之后客户端与服务器使用对称加密算法进行通信
